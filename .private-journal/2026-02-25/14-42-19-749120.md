Security findings to analyze carefully:

### Finding 1: transport_exec SSH command construction (lib/transport.sh lines 82-87)
```bash
ssh://*)
  local host="${target#ssh://}"
  local args=()
  for arg in "$@"; do args+=("$(printf '%q' "$arg")"); done
  _csd_ssh "$host" "${args[*]}"
  ;;
```
The individual args are quoted with printf '%q', but then joined with `${args[*]}` (space-joined string). This gets passed as a single string to ssh, which means the remote shell will re-split and re-parse it. The issue: `printf '%q'` produces bash-safe quoting but ssh passes the combined string to `sh -c` on the remote. If the target host is attacker-controlled (i.e., the user passes a crafted --target value), the `$host` variable is interpolated directly into the ssh command. But the more concerning issue is that `host` comes from `${target#ssh://}` which strips the prefix from whatever was passed as --target. This means if someone passes `--target ssh://; evil_command`, the semicolon would be... actually no, `_csd_ssh "$host"` passes host as a quoted argument to ssh, so that's fine.

But wait: there IS a subtle issue. `"${args[*]}"` passes the array joined by spaces as a SINGLE string argument to ssh. SSH then sends this as a shell command to the remote. The `printf '%q'` quoting uses bash-format quoting which should be interpreted by bash on the remote side... but if the remote shell is `/bin/sh` (not bash), the quoting might not be interpreted correctly. However, this is a functionality issue, not a security issue per se.

The WORKER_TARGET env var is loaded from a .meta file: `WORKER_TARGET=$(jq -r '.target // "local"' "$meta_file")`. The meta file lives in /tmp/claude-workers/, world-writable on most systems. If an attacker can write to /tmp/claude-workers/${session_id}.meta, they could set target to something malicious. But SESSION_ID is a UUID, so guessing it is hard. Not a critical finding but worth noting.

### Finding 2: SSH hook sync — unquoted mkdir call (launch-worker.sh lines ~375-376)
```bash
ssh "$HOST" "mkdir -p $REMOTE_PLUGIN_DIR"
```
`$REMOTE_PLUGIN_DIR` is set to `/tmp/csd-hooks-$SESSION_ID` where SESSION_ID is a UUID. This is unquoted inside the double-quoted string! If SESSION_ID contained special characters (it won't since it comes from uuidgen/python uuid4, which only produces hex+hyphens), this could be an injection. As SESSION_ID is always a UUID, this is low risk in practice, but technically unquoted.

The `scp` line:
```bash
scp -r "$PLUGIN_DIR/hooks" "$PLUGIN_DIR/.claude-plugin" "$HOST:$REMOTE_PLUGIN_DIR/"
```
`$HOST` comes from `${EFFECTIVE_TARGET#ssh://}` which strips the `ssh://` prefix from the user-supplied `--target` value. A crafted HOST value like `user@host -oProxyCommand=evil_cmd` could inject ssh options. This is a real injection risk if `--target` is user-controlled with malicious intent.

### Finding 4: transport_write docker — sh -c with filepath (lib/transport.sh line 142)
```bash
printf '%s' "$content" | docker exec -i "$container" sh -c "cat > $(printf '%q' "$filepath")"
```
The filepath is quoted with `printf '%q'`, which is correct bash shell quoting. However, `docker exec` runs `sh` (not necessarily bash) on the container, and sh may not understand `$'...'`-style quoting that bash's `%q` sometimes produces. This is more of a correctness/reliability issue, not a direct security problem.

### Finding 5: test mock ssh uses eval (tests/test-launch-worker-remote.sh lines ~1065, 1087)
```bash
eval "RCMD=(\$REMOTE)"
...
eval "\$REMOTE"
```
In the TEST file, the mock SSH binary uses `eval`. This runs in the test environment when the tests are executed. The `$REMOTE` variable comes from the arguments passed to the mock ssh script. In the test context, these are controlled by the test harness, so the eval of controlled input is as safe as tests can be. However, if this test were run against a malicious META file or with crafted inputs, the eval could be dangerous. But this is test code with mocked infrastructure, so this is a moderate concern.

### Finding 6: SSH ControlMaster socket path (lib/transport.sh lines 57-60)
```bash
ssh -o ControlMaster=auto \
    -o ControlPath="$HOME/.ssh/csd-%r@%h:%p" \
    -o ControlPersist=60 \
```
ControlMaster=auto with ControlPersist=60 creates a persistent SSH connection in ~/.ssh/. This is a common SSH optimization technique. The socket is named predictably using the remote user, host, and port. In a shared system, a malicious user who can create a socket at that path before a legitimate connection could intercept the connection (socket hijacking). On typical single-user development machines this isn't a practical concern. The 60-second persist means SSH connections stay open longer than necessary after use.

The ControlPath `~/.ssh/csd-%r@%h:%p` is fine, this is a standard pattern.

### Finding 7: WORKER_HOME from remote (launch-worker.sh line ~348)
```bash
WORKER_HOME=$(transport_exec sh -c 'printf "%s" "$HOME"' 2>/dev/null || echo "$HOME")
```
This runs `sh -c 'printf "%s" "$HOME"'` on the remote host and captures the output. The remote output is used as a filesystem path component later:
```bash
LOG_FILE="$WORKER_HOME/.claude/projects/${ENCODED_PATH}/${SESSION_ID}.jsonl"
```
If a malicious remote host returns a crafted value for $HOME (like `/../../../etc`), this could cause path traversal when reading log files. However, since the remote host is the one being controlled (it's a worker target), if the remote is malicious, there's bigger problems anyway. But this is worth flagging as a path traversal risk.

### Finding 8: Docker image name from --target (launch-worker.sh ~lines 329-332)
```bash
if [[ "$TARGET" == docker-run://* ]]; then
  IMAGE="${TARGET#docker-run://}"
  CONTAINER_NAME="csd-$SESSION_ID"
  docker run -d --name "$CONTAINER_NAME" "$IMAGE" sleep infinity
```
`$IMAGE` comes directly from the `--target` argument. An attacker with control over the `--target` value could run arbitrary docker images, which could be malicious. More importantly, `docker run` with a user-supplied image name is inherently a code execution primitive. This is the design intent of the feature, so it's not malicious, but it's a significant trust boundary consideration. The image name is passed as separate arguments to docker run, so no shell injection here.

### Finding 9: /tmp/claude-workers/ path (multiple files)
All session metadata, events files, and tool-decision files are stored in /tmp/claude-workers/. On Linux, /tmp is world-writable. Session IDs are UUIDs (hard to guess). The meta files contain the target URL, container_managed flag, worker_home path, etc. While TOCTOU attacks are theoretically possible, the UUID-based naming makes them impractical.

### Finding 10: Missing --no-sync-hooks documentation
The --no-sync-hooks flag is parsed but its security implications (trusting pre-existing remote hooks) aren't documented. Not a direct security issue.

2. MODERATE: WORKER_HOME path traversal (remote can return crafted $HOME value)
3. LOW: eval in test mock ssh (test code, controlled environment)
4. LOW: ControlMaster persistence (socket in ~/.ssh/)
5. LOW: printf '%q' compatibility with remote sh vs bash in transport_write docker
